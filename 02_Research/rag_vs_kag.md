# WikiGenius 与 RAG/KAG 的关系

> WikiGenius 不是 RAG，更接近 KAG，但也不完全是——它是一个 **KAG 的超集：知识一致性引擎 (Knowledge Consistency Engine)**。

---

## 概念定义

### RAG (Retrieval-Augmented Generation)

```
用户查询 → embedding → 向量相似度检索 → 召回相关文本块 → 塞进 LLM 上下文 → 生成回答
```

核心特征：
- **检索驱动**：靠向量相似度找"看起来相关的文本块"
- **只读**：不写、不改、不仲裁
- **无真值概念**：同一句话的两个矛盾版本，RAG 给你平均召回，不会告诉你哪个是对的
- **无逻辑推理**：相似度 ≠ 一致性。两段文本可以向量高度相似但逻辑完全矛盾

### KAG (Knowledge-Augmented Generation)

```
用户查询 → 实体/关系识别 → 知识图谱遍历 → 结构化事实召回 + 逻辑推理 → 生成回答
```

核心特征：
- **图谱驱动**：先建实体-关系-属性结构，查询时在图上推理
- **有逻辑一致性**：图结构天然支持"这个事实和那个事实是否矛盾"的推理
- **结构化的真值表达**：事实作为节点，之间有关系边，不是无结构的文本块

---

## WikiGenius 为什么不是 RAG

| RAG 特征 | WikiGenius 的实际行为 |
|----------|----------------------|
| 向量相似度检索文本块 | 不检索文本块，而是在知识图谱上评估**逻辑一致性** |
| 无真值模型，所有 chunk 权重相等 | 维护"当前公认真值"，标记过时信息 |
| 不检测矛盾 | 矛盾检测是核心价值主张——变更提交要过一致性校验 |
| 只读 | 读 + 写 + 仲裁 + 协商 |
| 权限无感知 | 权限模型嵌入知识结构（Space Admin / Project Owner / Member） |
| 无状态 | 有变更历史、冲突状态、冻结 section |

RAG 回答"这些内容在文档里"。WikiGenius 回答"这些事实在知识图谱里是否自洽"。

---

## WikiGenius 为什么是 KAG 的超集

### 和 KAG 的共同点

| KAG 特征 | WikiGenius 对应 |
|----------|----------------|
| 基于知识图谱而非向量索引 | Projects / Areas / Resources & Archives 三层知识结构 |
| 图遍历 + 逻辑推理而非相似度匹配 | 变更评估的一致性维度——在新事实和已有事实之间检测矛盾 |
| 结构化事实有明确关系 | 跨文档交叉引用、依赖关系、约束关系 |
| LLM 用于自然语言理解但推理靠图谱 | Team Agent 做一致性评估，用户通过自然语言交互 |

### WikiGenius 超出标准 KAG 的部分

| 能力 | KAG（典型） | WikiGenius |
|------|------------|------------|
| **写入 + 冲突检测** | 通常只做检索 + 生成 | 核心能力——提交变更、检测冲突、冻结 section |
| **多 Agent 协商** | 无 | User Agents 后台自主协商 |
| **权限-知识耦合** | 通常无权限模型 | 三层权限嵌入知识结构 |
| **真值维护** | 图谱通常是静态快照 | 持续更新的"当前公认真值"，包含变更生命周期 |
| **冲突仲裁链** | 无 | 协商失败 → Project Owners → Space Admin |

## 更精确的定位

标准 KAG 解决了"如何从知识图谱中更聪明地检索"——这是一个**读**问题。

WikiGenius 解决的是"如何在多人异步协作中保持知识图谱逻辑自洽"——这是一个**读写一致性**问题。它不仅要检索知识，还要在接受新知识时保证不被矛盾的知识污染。

这个"写入时的逻辑一致性校验"是 KAG 和 WikiGenius 之间的核心分界线。

```
          ┌──────────────────────────────────────┐
          │         WikiGenius                   │
          │  ┌────────────────────────────────┐  │
          │  │         KAG 的读能力            │  │
          │  │  (图谱推理 + 结构化事实检索)     │  │
          │  └────────────────────────────────┘  │
          │  ┌────────────────────────────────┐  │
          │  │        WikiGenius 独有          │  │
          │  │  • 写入时一致性校验              │  │
          │  │  • 多 Agent 冲突协商            │  │
          │  │  • 权限-知识耦合                │  │
          │  │  • 真值维护 + 仲裁链            │  │
          │  │  • 异步后台协商                 │  │
          │  └────────────────────────────────┘  │
          └──────────────────────────────────────┘
```

### 一句话定位

> WikiGenius = **KAG + 写入一致性 = 知识一致性引擎 (Knowledge Consistency Engine)**

如果要在学术界找一个对应概念，最接近的可能是 **Logic-Enhanced Knowledge Graphs + Multi-Agent Consensus**，但这是几个领域的交叉，没有一个现成的简写标签。

---

## 对技术选型的影响

这个定位意味着：

1. **不能用纯 RAG 方案**：向量数据库 + LLM 只能解决"搜"，不能解决"不矛盾"和"谁说了算"
2. **图数据库是必需的**：KAG 的底需要图结构来存储实体关系、交叉引用、约束关系
3. **但图数据库不够**：还需要一个**规则引擎/推理层**来评估"新事实是否和已有事实逻辑矛盾"——这是 WikiGenius 的技术硬核
4. **多 Agent 协商协议**是纯应用层的挑战，没有现成方案可以直接用
