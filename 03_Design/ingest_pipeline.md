# Ingest: Digest, Don't Pile

> **一句话.** 原始输入不会被原样堆进 wiki——User Agent 先把它**消化**成图能消费的原子事实，再作为"待评估变更"汇入 Team Agent 已有的三维校验。Ingest 是整条变更流水线的**前半段**，不是一个平行系统。
>
> **来源说明.** "消化而非堆砌"这个核心想法借自一个与 WikiGenius 无关的个人项目（单 Agent、本地 markdown、slash 命令）。本文**不照搬**它，而是按 WikiGenius 的双层 Agent + 知识图谱 + 一致性引擎重新解释。落地 [`design_readiness.md`](design_readiness.md) 的 **D2**，部分回答 D1/D4。

---

## 一、它在变更流水线里的位置

WikiGenius 的写入已经有后半段（Proposal 的「Change Evaluation」三维校验 + 「真值无损压缩」）。缺的是**前半段**：一篇乱糟糟的文章/日志/IM 讨论怎么变成图能评估的结构化变更。Decompose-Match-Merge 补的正是这一段：

```
原始输入（文章 / 日志 / 决策 / IM 线程）
        │
        ▼  ── 前半段：Ingestion（User Agent，Assistive AI）──
  1. Decompose  拆成原子 claim
  2. Match      每条 claim 在知识图谱里定位它的归属节点 + 相关/矛盾邻域
  3. Merge      提议把 claim 并入该节点（无匹配才新建）→ 产出"待评估变更集"
        │
        ▼  ── 后半段：已有机制（Team Agent，Coach + Diagnostic AI）──
  4. 三维校验   一致性 / 权限 / 影响范围（Match 已交出邻域，一致性校验几乎免费）
  5. 冲突       → 通知 + section 冻结 + 协商/升级（见 D7 协商协议）
  6. 接受       → 写入图谱与 Lark 载体 → 被取代检测 / 冗余合并 / 粒度压缩
```

**关键连接**：Merge 的产物是**变更提议，不是直接写入**。它走的是每条变更都要走的同一条评估通道。Match 不只是"找地方放"，它顺带交出了"这条 claim 的邻居和潜在矛盾"——而这正是第 4 步一致性校验最需要的输入。所以 ingest 和一致性校验是**同一条流水线的两端**，不是两套东西。

## 二、三步，用 WikiGenius 的话讲

**1. Decompose（User Agent）.** 把输入拆成原子 claim——每条只断言一件事。理由不是"为拆而拆"：一篇 postmortem 会同时触达 runbook、某条 decision、一个 metric；只有拆到 claim 粒度，每条才能被独立地定位、校验、归并，避免"整篇塞进一个文件夹"的错配。拆的同时执行**第一道克制**：一次性的、无复用价值的细节直接丢弃，不进入后续。

**2. Match（查图，不查本地索引）.** 每条 claim 在**知识图谱**里找三样东西：它要丰富的节点、它要更新的事实、它可能矛盾的已有 claim。检索靠图的主题/边结构 + 语义召回；在 Lark MVP 上，候选召回可直接用 `lark-cli docs +search`（跨 Docs/Wiki 搜索），再由 WikiGenius 的图层判定归属与矛盾。

> ⚠️ **与借来的范式的关键区别**：原项目用一个手工维护的 `knowledge-map.md` 当索引，因为它是一堆扁平 markdown、没有图。WikiGenius **有图**——图本身就是索引，再单独养一个 markdown 索引是冗余。若 Lark MVP 阶段需要一个查询缓存，可以用一张 **Base 表**作为图的**派生缓存**（可随时重建），但**真值在图里**，缓存不是源。

**3. Merge（提议，不是写入）.** 把 claim 并入匹配到的节点，丰富已有知识；**只有当没有任何节点匹配时**才新建节点。产物是带溯源标记的变更集（C4：标注"人类撰写 / User Agent 辅助 / 生成"），交给第 4 步评估。

## 三、`+` 操作 = Agent 行为，不是用户命令

借来的范式把 `+ingest/+promote/+audit/+status` 当成人敲的 slash 命令。WikiGenius 是双层 Agent 系统，所以这些是 **Agent 的内部行为**（部分可作为 Space Admin 的运维手柄暴露），按 AI 角色归位。它们恰好和 lark-cli 的 `+shortcut` 约定同形，落地时就是一组对 lark-cli 的编排。

| 操作 | 它其实是什么 | 归属 / AI 角色 |
|------|-------------|---------------|
| `ingest` | 上面的 Decompose-Match-Merge 前半段 | **User Agent**（Assistive AI） |
| `promote` | 内容跨 PARA 层成熟（Resource→Area、草稿→正式 Decision），过下面的准入门 | **Team Agent**（Coach AI） |
| `audit` | 扫描图找错位/老化/该并未并的内容 | **Team Agent**（Diagnostic AI），喂给压缩与衰减（C12） |
| `status` | 各 PARA 层节点数 + 膨胀趋势，是演示仪表盘和评估指标（D14）的数据源 | 任意；呈现在信任面（C6 可见性） |

> 没有保留原项目的 `+sync`——既然 Match 直接查图、缓存可随时重建，"刷新索引"不再是一个需要单列的仪式；它退化为图写入后的顺带维护。

## 四、Promotion Gate（跨层准入门，自有定义）

内容要从低成熟度层升到高成熟度层（更高权威、更被依赖）前，必须同时过三关——**不是**借来的那个品牌化的"Feynman Gate"，而是从 WikiGenius 自己的约束推出来的：

1. **可理解性（C11）.** 一个新成员只读这个节点能不能用起来？讲不清、靠行话堆砌的内容不许升级。（这恰好是"能否简单解释"那个直觉，但理由是 C11 共同理解，不是借来的名字。）
2. **不替用户裹装（C5）.** 升级只确认成熟度，不顺手润色语气、不为"显得正式"而扩写。
3. **权限签字（权限模型）.** 升级到 Area/正式 Decision 需要对应 owner 角色确认——把成熟度背书绑到一个具体的人（C10 问责），不是 Agent 自己拍板（C2 自主权）。

## 五、贯穿始终的克制（anti-bloat）

膨胀在三个点被拦截，而不是只靠事后清理：

- **Decompose 时**：丢弃一次性、无复用价值的 claim。
- **Merge 时**：能并入已有节点就不新建；同一事实的不同表述合并（与压缩的「冗余合并」同一套逻辑）。
- **维护时**：`audit` + 压缩负责降级老化内容（C12 衰减）。

判定一条 claim 去留的启发式：**它会改变某个未来决策吗？** 会则留，纯记录则弃，与已有事实矛盾则不算膨胀问题、转入冲突评估。

## 六、与学术约束的对齐

| 约束 | 本流水线如何满足 |
|------|------------------|
| **C5** 不优化用户没要的目标 | Decompose/Merge 只提取与归并 claim，不润色；Promotion Gate 拒绝裹装 |
| **C4** 溯源 | Merge 产物带"人类/Agent"来源标记，写入节点 provenance（D4） |
| **C2** 人类自主权 | Merge 只产出*提议*，结构性变更/冲突/跨层升级都路由给人 |
| **C11** 共同理解 | Promotion Gate 的可理解性关，保证图能让新人快速上手 |
| **C12** 知识衰减 | `audit` + 压缩降级老化内容 |
